# 项目时间线与风险评估

## 文档用途

这份文档用于明确：

- 从当前进度到完整项目原型，大致需要多久
- 各阶段的合理时间预估
- 项目推进中最容易导致延期的风险点
- 实时链路与语音链路中的延迟风险

本文档面向工程推进和项目节奏控制，不是实现细节文档。

---

## 一、整体时间判断

按当前完成进度与项目目标评估，Asurada 若要做到“第一版完整可交付原型”，比较现实的时间区间是：

- 阶段一完全封板：`1 - 2 周`
- 阶段二完成可用版：`6 - 10 周`
- 阶段三完成产品化原型：`8 - 16 周`
- 整体第一版完整原型：`4 - 7 个月`

这里的“完整原型”指的是：

- 真实数据链
- 规则策略脑
- 阶段二模型
- 双向语音原型
- 本地部署能力
- 产品化前的稳定运行原型

如果后续继续扩：

- 更多赛道
- 更多 session 类型
- 更复杂联机支持
- 更强语音交互
- 更深硬件联调

则整体周期会继续拉长。

## 二、阶段时间线

### 阶段一：核心链路封板

目标：

- 真正封板
- 明确边界
- 做完协议精修尾项
- 明确哪些内容进入阶段二

当前预计剩余时间：

- `1 - 2 周`

主要工作：

- `session_type` 继续收口
- `LobbyInfo` 验证准备
- 样本回归收口
- 封板文档
- 剩余协议边界整理

### 阶段二：模型研发与旁路接入

目标：

- 建立稳定训练数据管线
- 完成第一批基线模型
- 做规则版与模型版对比
- 让模型以旁路方式接入系统

预计时间：

- `6 - 10 周`

建议节奏：

- 第 `1 - 2` 周：
  - 特征导出
  - 标签体系
  - 数据切分
- 第 `3 - 4` 周：
  - 风险评分模型 baseline
  - 动作排序模型 baseline
- 第 `5 - 6` 周：
  - 驾驶动态评分模型
  - dashboard 接入
- 第 `7 - 8` 周：
  - 回归
  - 调优
  - 特征消融
- 第 `9 - 10` 周：
  - 阶段二验收
  - 语音原型旁路接入

### 阶段三：实时部署与产品化原型

目标：

- 本地实时推理
- 双向语音正式接入
- Pi 5 / CM5 优化
- 系统降级与可靠性增强

预计时间：

- `8 - 16 周`

建议节奏：

- 第 `1 - 3` 周：
  - live UDP 主链
  - 本地推理主链
- 第 `4 - 6` 周：
  - 本地 ASR / TTS
  - 双向语音链路
- 第 `7 - 10` 周：
  - 边缘设备优化
  - 延迟测试
  - 降级机制
- 第 `11 - 16` 周：
  - 长时稳定性
  - 产品化收口
  - 硬件联调

## 三、项目延期的主要风险

### 1. 数据样本不足

风险描述：

- 不同赛道样本不够
- 不同 session 类型样本不够
- 联机样本不足
- 极端天气 / 特殊事件样本不足

影响：

- 模型泛化差
- 协议语义难完全定版
- 很多模型只能停留在“局部可用”

严重程度：

- `高`

缓解方式：

- 优先积累真实比赛样本
- 按 `session_uid` 建样本库
- 保留 metadata 和验证脚本

### 2. 标签体系不稳定

风险描述：

- 风险标签定义反复变化
- 策略动作标签不一致
- 驾驶评分标准不统一

影响：

- 所有训练结果都可能返工
- 模型对比没有意义

严重程度：

- `高`

缓解方式：

- 先定义标签字典
- 区分自动标签与人工校验标签
- 标签版本化

### 3. 同时推进过多模型

风险描述：

- 同时铺太多方向：
  - 风险评分
  - 动作排序
  - 驾驶评分
  - 胎耗预测
  - 对手态势
  - 解释层

影响：

- 每条线都推进不深
- 阶段二收不了口

严重程度：

- `高`

缓解方式：

- 阶段二只聚焦 `2 - 3` 个核心模型
- 先做高价值 baseline

### 4. 低估双向语音复杂度

风险描述：

- 误以为“ASR + TTS”就等于双向语音

实际难点：

- 背景噪声
- 意图识别
- 中断与抢占
- 主链与解释链分离
- 高压驾驶场景下的短句回答

影响：

- 阶段三容易明显延期

严重程度：

- `高`

缓解方式：

- 先做单向播报
- 再做结构化问答
- 解释层保持旁路

### 5. 过早进入边缘部署优化

风险描述：

- 在模型和语音能力还没稳定前，就开始做 Pi 5 / CM5 优化

影响：

- 大量时间花在性能和硬件细节
- 核心算法和产品逻辑反而没收稳

严重程度：

- `中高`

缓解方式：

- 阶段二先在开发机完成验证
- 阶段三再集中做边缘优化

### 6. 协议边界比预期复杂

风险描述：

- 某些 packet 需要更多样本才能彻底确认
- 某些 session type 语义需要继续收口

影响：

- timing、gap、多人联机、特殊事件语义收尾时间拉长

严重程度：

- `中`

缓解方式：

- 明确“项目内稳定命名”和“官方最终命名”分层
- 不等待所有细节完全定版再推进阶段二

## 四、实时链中的延迟风险

### 1. UDP 到状态仓延迟

风险来源：

- packet 解码耗时
- 组帧等待
- 丢包和缺包处理

影响：

- 策略判断滞后
- 语音播报时机错误

缓解方式：

- 简化主链 packet
- 缺包降级
- 主链数据结构固定化

### 2. 模型推理延迟

风险来源：

- 模型太大
- 特征预处理太重
- 多模型串行调用

影响：

- 主路径超时
- 语音响应变慢

缓解方式：

- 主路径只保留小模型
- 解释层旁路
- 模型分层：主路径 / 旁路

### 3. ASR 延迟

风险来源：

- 本地识别模型太重
- 流式识别不稳定
- 背景噪声太大

影响：

- 车手语音查询响应慢

缓解方式：

- 优先本地流式 ASR
- 热词优化
- 结构化命令优先

### 4. TTS 首音延迟

风险来源：

- TTS 初始化慢
- 文本太长
- 音频缓冲过大

影响：

- 策略已经出来，但播报晚

缓解方式：

- 用短模板
- 首音优先
- 支持打断与覆盖

### 5. 大模型解释层阻塞

风险来源：

- 把解释层放进主路径

影响：

- 双向语音不可实时
- 整个输出链卡住

缓解方式：

- 大模型解释层必须旁路
- 主路径只允许结构化回答

### 6. 消息仲裁延迟

风险来源：

- 消息队列太长
- 优先级策略复杂
- 不支持打断

影响：

- 低价值消息占满播报窗口
- 高优先级消息说晚了

缓解方式：

- 高优先级抢占
- 低优先级冷却
- 关键区段少说话

---

## 五、当前最现实的推进策略

为了把延期风险压低，建议遵循以下节奏：

1. 阶段一尽快封板
2. 阶段二只聚焦 `3` 个核心模型
   - 风险评分模型
   - 动作排序模型
   - 驾驶动态评分模型
3. 双向语音先做结构化问答，不急着做开放式解释问答
4. 阶段三先稳定单向语音，再接完整双向语音
5. 边缘设备优化放到阶段三集中做

---

## 六、一句话判断

如果控制范围，不同时铺太多线，Asurada 做到第一版完整可交付原型，大约需要：

- `4 - 7 个月`

最大的延期风险不是“写不出来代码”，而是：

- 数据和标签不稳定
- 模型方向铺太大
- 低估双向语音实时难度
- 过早进入边缘部署优化
